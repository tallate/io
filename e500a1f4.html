微服务的相对优势


图中，小圆形、方形、三角这些小图形是服务实例，包围小图形的双层方形是Docker容器实例，立方体是物理服务器（一般是云提供商提供的虚拟机实例）
微服务应用相对单体式应用的优势体现在下面这些方面：

  1. 可扩展性高
    严格界定服务边界，服务之间是弱耦合的，每个服务本身是无状态的，可以通过水平复制和负载均衡来提高服务性能。
  2. 实施效率高
    应用Docker实现运维自动化，且每个服务都足够小，使得部署效率不会太低。
  3. 健壮
    某个服务的不可用不会引起大规模雪崩，而且服务的复制也提高了整体的可用性。
  4. 单个服务复杂性低
    每个服务都足够简单，由一个独立团队来负责。

Web客户端部署在哪？在Nginx服务器上。
Redis那些呢？因为空间关系省略了。

RabbitMQ多此一举？RabbitMQ可以解耦消息管理服务和业务服务，另外，消息队列也是实现异步调用必须的，消息队列提供重试、，有利于实现可靠消息的最终一致性效果。

  1. 分布式锁：保证临界区内互斥
  2. 版本号：乐观锁

描述更新操作的主要流程：

  1. 在后台接收请求后需要先为实体的主键添加分布式锁，分布式锁主要用于防止两个线程同时进入临界区

  2. 版本号的作用是避免并发的修改操作，或者说修改旧的数据

  3. 分布式锁的设计问题，有考虑到加锁操作的原子性、只有自己可以解锁自己、死锁、线程宕掉、耗时、失效时间、阻塞、可重入？

  4. 版本号能不加吗？不行，因为业务上后续可能还要进行融资分析等，需要提高安全性


  1. 取放款单中的开始日期、结束日期、结息日得到每个月的日期列表，比如1.1、4.1、25号结息，则得到1.25、2.25、3.25
  2. 根据结息单位过滤，比如如果是按季度结息的，则过滤得到3.25
  3. 根据结息周期过滤，比如如果周期是2，则每隔一个日期取一条
  4. 如果结束日期不在日期列表里，则将结束日期添加进来
  5. 如果是计算还本还款计划，则每个日期需要付款的金额是放款金额除以日期个数，如果除不尽则加到最后一个日期上；如果是计算付息还款计划，则每个日期需要付息的金额是利率 x 放款金额 x 距离上一个日期点的天数。



思路
说明提交操作是需要跨服务写库的，除了在修改操作的加锁和版本校验之外，还需要注意TCC的执行流程：

  1. 解释TCC事务需要每个参与者提供try、confirm和cancel三个接口，try执行资源的预留，比如对涉及的实体加锁，也起着服务验活的功能，confirm执行事务的提交，也就是真正地执行业务操作、将数据持久化到数据库，cancel需要释放预留的资源；
  2. 调用ID和链路ID的作用；
  3. 每个事务参与者将自己的信息提交到Redis缓存内，之后根参与者需要取得所有参与者信息，主要是它们的confirm和cancel接口的地址；
  4. ①如果业务操作都成功了，由第一个参与者来调用所有其他参与者的confirm接口来提交事务；
    ②如果业务操作失败了，同样由第一个参与者来调用所有其他参与者的cancel接口来回滚事务。

提问点

  1. TCC没讲明白？用一个常见的用户购买商品场景为例，需要先确保用户能够支付，然后锁住用户和商户的账户，这是try阶段；如果try成功，即预留资源成功了，接下来再对用户账户扣款、商户账户入账，如果因为超时等原因失败了，需要事务补偿，即重试，这是confirm阶段；如果try失败，则释放所有的锁，这是cancel阶段。
  2. 事务补偿时如何保证幂等性？
  3. 本地事务如何处理（对MySQL的理解）？

总结与展望

—-总结
学习分布式理论基础

  1. 学习进程、虚拟化和服务等概念；
  2. 学习通信的常用方法；
  3. 学习同步和一致性的概念。

解决系统设计中的难点

  1. 同步服务发现服务器和网关服务器，实现服务弹性伸缩；
  2. 设计基于HTTP的RPC协议，实现服务间通信；
  3. 使用分布式锁、版本控制、TCC事务来保证服务间一致性。

实现系统

  1. 后台除注释外代码达到近2万行，类近500个；
    2.独立数据库5个，数据库表共18张；
  2. 系统包含相互独立的微服务6个。

—-改进
服务发现

  1. 使用Eureka等高可用服务发现产品替代ZooKeeper；
  2. 使用复制和DNS等手段工具提升Nginx服务器的可用性。

服务间通信

  1. 使用TCP等更底层、灵活的协议；
  2. 使用Kryo等更完善的序列化工具。

服务间一致性

  1. 做好TCC事务的异常日志，供出错时手动执行恢复；

思路

  2. 介绍论文里主要解决的三个难点
    提问点
  3. 概念，Zab协议、选举、容错集群、Eureka、Kyro、redo/undo日志。
  4. ZooKeeper集群通过Zab协议保证高一致性，重新选举Leader比较耗时，且当节点数量不足以构成容错集群时，ZooKeeper倾向于返回空；故提出改用Eureka等其他倾向于提供高可用性的具有服务发现功能的产品；
  5. Nginx服务器作为网关，存在单点故障风险，但是这里的服务发现只能保证被Nginx代理的那些服务能弹性伸缩，对Nginx本身却没有什么办法，就算复制了Nginx服务器，客户端也只能发到一个IP上。这里提出使用基于操作系统的方案对Nginx集群实现验活及负载均衡，主要是Keepalived等软件或独立的DNS服务器，但是论文题目限于Saas层展开，本人水平有限没法解释；
  6. HTTP是应用层的，Spring Boot封装了一个RestTemplate可以很方便地发HTTP请求。而TCP是传输层的，如何包装消息需要另外再讨论，常见的框架如Netty，提供了长连接、心跳检测等功能，可以提高开发效率；
    Kryo是一个优秀的Java序列化方案，大概用了很多优化方案，这就是很零碎的编程问题了；
    TCC的confir和cance都有可能出错，一般都需要手动恢复，异常日志指的是redo和undo日志，redo是未执行前的数据值，undo是执行修改后的数据值。
