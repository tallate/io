¶一面

  1. 接口能加什么修饰符？
    A. private
    B. protected
    C. final
    D. abstract
    接口本身就是抽象的，加不加abstract都一样。private和protected违反了接口需要开放的原则、实际上外部类和接口都只能用public或默认修饰，如果加了final接口，接口不能被实现了（final的语义），那接口几乎也没意义了，总之这些修饰符都是不合法的。

  2. 给出下面代码的输出

    字符串字面量会被保存到方法区（常量池），而直接调用构造器构建的字符串对象则会分配到堆上，主要问题出在d和e上，d是两个字符串的连接，编译器会暗中使用StringBuilder完成这个任务，e中两个字符串字面量的拼接会被优化为一整个字符串（常量优化）。因此，其输出应该是下面这样的：
    true
    false
    false
    true

  4. Object类中有哪些方法？写出它们的作用。
    clone()：复制对象。
    getClass()：获取对象所属的Class。
    hashCode()：计算hash值，默认为地址。
    equals()：比较对象是否相等，默认为比较地址。
    toString()：默认情况下结果为"类名@地址"。
    wait()：挂起当前线程，如果没有占有该对象上的监视器锁则抛出异常。
    wait(long timeout)：如果没有在timeout时间内被其他线程调用该共享对象的notify()或notifyAll()方法唤醒，则该函数仍会因为超时而返回。
    wait(long timeout, int nanos)：如果nanos>0则让timeout参数递增1。
    notify()：唤醒一个在该共享变量上调用wait方法挂起的线程，如果有多个线程则唤醒的目标是随机的。如果没有占有该对象上的监视器锁则抛出异常。
    notifyAll()：唤醒占有该对象监视器锁的所有线程。
    还有一些不常见的：
    finalize()：在对象被回收时调用。
    registerNatives()：用于注册native方法。

  5. 写一个单例模式
    当时直接用枚举enum写了个单例。

    最常见的单例模式实现方式包括懒汉式和饿汉式，懒汉式的实现还需要注意线程安全性，需要考虑使用synchronized还是双检锁；静态内部类方式可以实现延迟初始化但是实例域必须是静态的；枚举类方式实现起来比较简单，具有线程安全性，自动支持序列化（可以看反编译字节码后的代码），但是没有延迟初始化的特性。

  11. JVM垃圾回收过程。
    GC分为检测和清理两个过程，检测过程比较有代表性的算法是可达性分析算法，清理过程有代表的是分代回收算法。
    可达性分析算法会先对所有可以从Roots到达的节点作标记，然后回收掉所有没有标记的对象。
    分代回收算法会将堆空间分成年轻代和老年代。新对象会被放到新生代的Eden区，当达到回收条件时，会触发新生代GC（复制算法），将有引用的复制到新生代的Survivor区，原有的Survivor中有引用的也会被复制到新Survivor中。在这个过程中对象的年龄会增长，年龄足够大后对象会被转移到老年代。当老年代达到回收条件时，会触发老年GC（标记整理或标记清除）。

  4. 死锁产生的原因是什么？（提示我从操作系统的角度来回答）
    以两个酒鬼相遇的场景为例，酒鬼A手里拿着开瓶器，B手里拿着啤酒。它们之间死锁的产生有以下4个条件：

    1. 互斥条件：啤酒、开瓶器在一段时间内只能由一个进程占用；
    2. 请求和保持条件：每个酒鬼都非常贪婪，他们不会放开自己手上的资源，并且渴求对方手上的资源；
    3. 不剥夺条件：他们比较怂，没胆子直接从对方手上把东西抢过来；
    4. 循环等待条件：他们陷入了一个怪圈，导致谁也喝不到啤酒。

  5. 能做个工具自动发现死锁的存在吗？动态分析（数据库、操作系统、用户程序）？静态分析（编译器）？
    这个问题应该考的是操作系统、数据库、虚拟机（应用程序）中都存在的死锁问题。
    在操作系统中，为了限制进程对临界区的并发访问，一般会对资源进行加锁，但是对资源的竞争或进程推进顺序的不当都有可能会引起死锁，为了处理死锁，一般有四种方法：

    1. 预防死锁。破坏死锁4个必要条件中的某一个，但是施加的限制太严格会导致系统资源利用率和系统吞吐量降低。
    2. 避免死锁。不破坏死锁条件，而是在资源的动态分配过程中，用某种方式去防止系统进入不安全的状态（安全状态下每个进程都能被分配到其所需的资源），从而避免发生死锁。
    3. 检测死锁。不检测系统是否进入不安全区，而是允许系统在运行过程中发生死锁，通过某种检测机制来检测死锁的发生，并精确地确定与死锁有关的进程和资源，然后采取适当措施，从系统中将已发生的死锁清除掉。
    4. 解除死锁。这种方法需要和检测死锁搭配，当检测到系统中已经发生死锁时，撤销或挂起一些进程，以便回收一些资源，再将这些资源分配给已处于阻塞状态的进程，使之转换为就绪状态，以继续运行。
      在数据库中，锁又叫封锁（Blockade），封锁的主要用处是同步多个事务对数据对象的修改，目前数据库中解决死锁的方法主要有两类：死锁预防、死锁诊断与解除，死锁预防主要有一次封锁法和顺序封锁法两种，死锁诊断方法主要有超时法和事务等待图法两种，因为效率方面的问题，数据库中不适合预防死锁，只适合进行死锁的诊断与解除。
    5. 一次封锁法。每个事务将所有要使用的数据全部加锁。这种方法由于增加了锁的粒度，从而降低了系统的并发性，而且事先很难精确地确定每个事务所需进行加锁的对象，为此只能扩大封锁范围，将事务执行过程中可能需要封锁的数据对象全部加锁，导致进一步降低了并发度。
    6. 顺序封锁法。按顺序进行加锁，比如如果索引是B+树结构的，可规定封锁的顺序必须是从根节点开始的，然后一层一层逐级封锁。但是这种方法也需要事先知道一个事务需要封锁哪些对象，面临和一次封锁法相同的问题。
    7. 超时法。如果一个事务的等待时间超过了规定的时限，那么就认为其发生了死锁。这种方法可能会误判，而且时限设置得太长则不能及时发现死锁。
    8. 事务等待图法。事务等待图中每个节点表示一个事务，每条边表示等待情况，并发控制子系统周期性生成事务等待图并检测，如果图中存在回路则表示系统中出现了死锁，需要执行死锁解除。
    9. 解除死锁。DBMS一旦检测到系统中存在死锁，就会从中选择一个处理死锁代价最小的事务，将其撤销，释放此事务持有的所有的锁，使得其他事务得以继续运行下去。当然，对撤销的事务所进行的数据修改必须加以恢复（可以用bin log恢复）。
      虚拟机如JVM，更准确地说应该是建立在JVM之上的JDK，提供了一些锁工具供用户同步多线程的临界代码，最常用的锁有四种：synchronized（同步锁）、ReentrantLock（可重入锁）、ReentrantReadWriteLock（读写锁）、StampedLock（戳锁）。其他的还有CyclicBarrier、CountDownLatch、Semaphore。平时研究一些同步问题（读者写者、生产者消费者等）时可以借助这些工具。在运行Java程序时如果发生死锁，程序会卡住，类似的现象还发生在IO、线程wait、sleep等情况时，但是它们的运行时堆栈是不一样的，一般可以用jstack来排查具体起因。如果是C语言则需要借助pstack和gdb这些工具，这篇博客有介绍。
      如果要进一步研究动态检测方法，必须要深入到Linux、MySQL源码中去了，现在就此先打住。
      对静态分析这一块我只有大学时上编译原理留下的一点印象作为基础，当面试官提到静态分析时我当然答不上来了，最后随便给了个莫名奇妙的答案，不大满意。事后上网冲浪（搜索关键词“静态检测 死锁”，Google“static analyze deadlock”）发现这一块研究其实已经很多了。并且在分布式系统中也有很多研究，因为分布式系统中的死锁更复杂。这份PPT介绍得比较好。

  6. Java中异常有哪些？Exception和Error区别？RuntimeException的含义？Exception有哪些别的子类？
    Error代表JVM出错，Exception分RuntimeException和其他异常，RuntimeException属于程序员犯了错，比如代码里出现除以零的表达式，所以是非捕获异常，而其他异常一般是应用程序所处的运行环境出了问题，比如网络异常，这种情况下就需要考虑进行重试或者恢复了。

  7. C语言中没有异常捕捉机制，让你实现怎么做？以C语言中除以0为例。
    C语言不支持异常处理，那么C++、Java是怎么实现异常处理的呢？主要有两种方式：goto和setjmp()+longjmp()。goto有很多缺点，后者的大概原理是保留调用处的环境信息（调用栈），在调用异常处理函数后再还原回来。
    比如socket是调用了操作系统接口，如果返回值错误则可以抛出异常，跳转到异常处理函数中。
    但是除0异常属于OS下面的计组的范畴了，当执行1 / 0对应的指令时，会引发处理器的0号中断。如果真的写了1 / 0这样的代码，编译器语义分析是能检测出来的，但只会给出一个警告。

  9. 乐观锁是什么？
    项目中使用到的版本号和时间戳都属于乐观锁。
    乐观锁是一种思想，其核心是只在更新的时候对数据是否产生并发冲突进行检测，如果发现并发冲突了，则返回错误信息。
    CAS是实现乐观锁的一种技术，因为太常用所以很多时候CAS也指代乐观锁。CAS操作中只有一个线程可以更新变量的值，而其他线程都失败，失败的线程并不会被挂起，而是被告知这次竞争中失败，并可以再次尝试。Java中的JUC包是建立在CAS之上的。

    CAP是分布式系统的三个主要特征：一致性、可用性、分区容错，CAP是不能同时满足的，一般倾向于实现CAP的妥协退化版BASE，经典的做法是实现基于消息队列的可靠消息事务（消息重发、接口调用原子性、与本地事务的配合）。
    Paxos协议的目的是在多副本中确定唯一值。分布式系统的一个集群中不能存在多个Leader，否则会出现脑裂现象（用多个Leader指挥一个小组为例）。如何确定唯一Leader？选举算法。确定唯一Leader后如何发送数据？消息广播。这两个过程的基础都是Paxos协议，只是实现细节可能会有所不同，比如ZooKeeper中的ZAB协议就是从Paxos延伸而来的。
